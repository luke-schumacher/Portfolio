Customer Digital Twin

Siemens Healthineers \u2022 Ongoing

Technical Architecture

Implemented a Conditional Variational Autoencoder (CVAE) architecture for generating synthetic customer profiles in healthcare technology sector. The system uses a dual-encoder design to capture both demographic and behavioral features, with a latent space dimensionality of 128.

  • Model: PyTorch-based CVAE with KL divergence regularization
  • Input: 47 features across customer demographics, purchase history, and engagement metrics
  • Latent Representation: 128-dimensional continuous space
  • Output: Personalized engagement strategy recommendations
  • Infrastructure: Deployed on AWS EC2 with S3 data storage

Challenges & Constraints

  • Data Privacy: Operated under strict GDPR and HIPAA compliance requirements
  • Class Imbalance: Healthcare customer segments highly skewed toward specific profiles
  • Interpretability: Need to explain model decisions to non-technical stakeholders
  • Real-time Performance: Required <200ms inference time for production deployment

Solutions & Trade-offs

Sampling Strategy: Implemented SMOTE-based oversampling in latent space rather than input space, preserving feature correlations while addressing imbalance.

Interpretability: Added attention mechanism to decoder, generating feature importance scores for each recommendation. Trade-off: 15% increase in inference time, but critical for stakeholder buy-in.

Performance Optimization: Quantized model to INT8 using TensorRT, achieving 3x speedup with <2% accuracy degradation.

Team & Role

Team Size: 4 engineers + 2 domain experts
My Role: Lead ML Engineer - Designed architecture, implemented training pipeline, optimized for production deployment, and presented findings to executive stakeholders.

Key Learnings

  • Technical: Learned importance of latent space analysis before deployment - discovered customer clusters not visible in input space
  • Process: Early stakeholder involvement critical - weekly demos prevented late-stage requirement changes
  • Defense/Production Relevance: Experience with compliance, interpretability, and performance optimization directly applicable to mission-critical systems requiring auditability and real-time response
Master's Thesis - Ongoing

Agentic Infrastructure Co-Pilot

Multi-Agent System for Fault Diagnosis

Technical Architecture

Distributed multi-agent system (MAS) combining Retrieval-Augmented Generation (RAG) with Knowledge Graph technology for autonomous infrastructure fault diagnosis. Three specialized agents collaborate through a coordinator:

  • Governance Agent (Telekom): Interprets network intent from policy documents (PDFs) using semantic search
  • Hardware Agent (Siemens): Diagnoses physical equipment faults from technical manuals (CSV/PDF)
  • Telemetry Agent (Illigo): Monitors event logs (JSON) for anomaly detection
  • Knowledge Graph: Neo4j graph database linking devices, errors, procedures, and events
  • RAG Pipeline: ChromaDB vector store + GPT-4 for grounded reasoning
  • Orchestration: FastAPI microservices with message queue (RabbitMQ)

Challenges & Constraints

  • Data Heterogeneity: Integrating 3 distinct data sources with different formats, schemas, and update frequencies
  • Agent Coordination: Preventing conflicting diagnoses while maintaining agent autonomy
  • Scalability: System must handle 10,000+ devices with real-time telemetry streams
  • Evaluation: No ground-truth labels for complex multi-domain faults

Solutions & Trade-offs

Data Integration: Built custom parsers for each source + unified entity resolution layer. Trade-off: Added latency (avg 200ms) but ensured data consistency.

Agent Coordination: Implemented voting mechanism with confidence weighting. Agents propose diagnoses + confidence scores; coordinator aggregates using Dempster-Shafer theory.

Evaluation Metric: Introduced "Mean Time to Innocence" (MTTI) - time to correctly identify fault is NOT in specific domain. Validated against subject matter experts in simulation.

Team & Role

Team Size: Solo thesis project with 3 external advisors
My Role: End-to-end system design, implementation, and evaluation

Key Learnings

  • Technical: RAG retrieval quality is bottleneck - invested 40% of time optimizing chunking strategy and embedding models
  • Systems Thinking: Agent specialization more effective than single monolithic model - enables parallel processing and domain expertise
  • Defense Applicability: Multi-agent architecture directly applicable to C4ISR systems requiring distributed decision-making under uncertainty

Prompt Academy

Agent-Based Educational Platform

Technical Architecture

  • Frontend: Vanilla JavaScript with HTML5 Canvas for interactive game UI
  • LLM Integration: Groq SDK (Llama-3.1-70B-Versatile) for real-time agent responses
  • Agent Design: 5 distinct character agents with unique system prompts and personalities
  • State Management: Client-side game state with localStorage persistence
  • Deployment: Static site on Netlify with edge functions for API key management

Challenges & Constraints

  • Prompt Injection: Players might try to "break" the agents by revealing system prompts
  • Response Variability: LLM non-determinism could make progression unpredictable
  • Latency: API calls averaging 2-3 seconds impacted game flow
  • Cost: Budget constraint of $50/month for educational project

Solutions & Trade-offs

Robustness: Implemented multi-layer prompt defense: system prompt reinforcement, meta-prompting to detect jailbreak attempts, and fallback responses. Trade-off: Slightly reduced agent spontaneity but improved reliability.

Response Validation: Added lightweight classifier to validate agent responses before display, retry on failures. Reduced progression variance by 60%.

Cost Optimization: Caching common responses + rate limiting. Kept within budget while serving 200+ users.

Team & Role

Team Size: 3-person student team
My Role: Technical Lead - Designed agent architecture, implemented LLM integration, and built game mechanics

Key Learnings

  • Prompt Engineering: System prompt clarity more important than length - concise, directive prompts outperformed verbose ones
  • User Testing: Players found creative exploits we never anticipated - importance of adversarial thinking in AI system design
  • Defense Relevance: Experience hardening LLM agents against adversarial inputs applicable to secure AI assistants and chatbot systems
Production System - Siemens Healthineers

MRI Scheduling AI

Transformer-Based Predictive Model

Technical Architecture

  • Model: Custom Transformer architecture with 6 encoder layers
  • Input Sequence: Historical machine usage patterns (30-day rolling window)
  • Features: 15 engineered features including time-of-day, day-of-week, patient priority, procedure type
  • Output: Next 7-day usage probability distribution per machine
  • Framework: TensorFlow 2.x with Keras API
  • Deployment: TF Serving on Kubernetes cluster

Challenges & Constraints

  • Data Quality: Missing data during maintenance windows, irregular sampling
  • Temporal Dependencies: Multi-scale patterns (hourly, daily, weekly cycles)
  • Production Requirements: Must achieve >85% accuracy to impact operations
  • Inference Speed: <50ms latency for real-time scheduling decisions

Solutions & Trade-offs

Data Imputation: Implemented time-series specific imputation using linear interpolation for short gaps, seasonal decomposition for long gaps. Preserved temporal patterns better than mean-filling.

Multi-Scale Attention: Added positional encoding with multiple time scales. Captured both short-term (hourly) and long-term (weekly) patterns. Trade-off: 30% more parameters but 12% accuracy improvement.

Model Compression: Applied knowledge distillation to reduce model size by 4x while maintaining accuracy within 1% of original. Met latency requirements on standard hardware.

Team & Role

Team Size: 5-person ML team + 2 domain experts
My Role: ML Engineer - Designed transformer architecture, built training pipeline, conducted ablation studies, and deployed to production

Key Learnings

  • Technical: Transformers excel at long-range dependencies but need careful pos positional encoding design for time-series vs NLP
  • Process: Close collaboration with radiologists revealed insights (e.g., emergency cases) that dramatically improved model
  • Production Impact: Achieved 89% accuracy, enabling 15% improvement in resource utilization - directly applicable to logistics and operations optimization in defense contexts
